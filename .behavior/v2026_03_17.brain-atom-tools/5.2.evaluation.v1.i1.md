# 5.2.evaluation.v1

## summary (as implemented)

implemented tool use support for rhachet-brains-chutes brain atoms, mirrored from the rhachet-brains-xai implementation. the implementation added:

1. **cast functions** — convert between rhachet and openai tool formats (3 files)
2. **ask signature extension** — added TPlugs generic and tool-aware prompt type
3. **tool call extraction** — detect and convert tool_calls from api response
4. **tool continuation** — support prompt as tool execution results

the pattern follows openai-compatible function call api, which chutes supports via vllm backend.

### divergence resolution

| divergence | resolution | rationale |
|------------|------------|-----------|
| [case7] omitted | backup | chutes api has mutual exclusivity between `response_format: { type: 'json_schema' }` and `tools` parameter. when tools are plugged, the api call includes tools parameter but cannot also include response_format with json_schema. this means the model may return unstructured text, not parseable JSON. the code handles this (lines 128-142: `shouldUseResponseFormat = !hasTools || isToolContinuation`), but a test would be flaky since we cannot force json output with tools present. this limitation is documented in the earlier review: `.behavior/v2026_03_17.brain-atom-tools/review/self/3.3.1.blueprint.product.v1.2.has-questioned-assumptions.md`. the usecase.3 criteria is covered by the code path but not testable reliably. |

**implementation matches blueprint with one documented divergence.**

- 3 cast files created: exact match
- genBrainAtom.ts modified: all codepaths implemented

the divergence does not affect the implementation — the code handles the case where brain answers directly with tools plugged (lines 244-254). the limitation is in test coverage, not behavior.
