# 1.vision: tool use for rhachet-brains-chutes

## the outcome world

### what does a day-in-the-life look like with this in place?

a developer building an ai-powered application uses chutes.ai for cost-effective inference. today they can ask the brain simple questions and get structured responses. after this feature, they can plug tools into the brain — weather lookups, calculators, database queries, api calls — and the brain decides when to invoke them.

```ts
const atom = genBrainAtom({ slug: 'chutes/qwen3/coder-next' });

// before: brain can only answer from knowledge
const result = await atom.ask({
  prompt: 'what is 42 * 17?',
  schema: { output: z.object({ answer: z.string() }) },
});
// result.output.answer = "714" (brain computes in head, may be wrong)

// after: brain can call tools
const result = await atom.ask({
  prompt: 'what is 42 * 17?',
  plugs: { tools: [calculatorTool] },
  schema: { output: z.object({ answer: z.string() }) },
});
// result.calls.tools = [{ slug: 'calculator.multiply', input: { a: 42, b: 17 } }]
// result.output = null (brain wants to use tool first)
```

### what's the before/after contrast?

| before | after |
|--------|-------|
| brain answers from internal knowledge only | brain can invoke external tools |
| structured output always populated | output null when tool invocation requested |
| no `calls` in response | `calls.tools` array with invocations |
| prompt is always a string | prompt can be tool execution results |
| single-turn inference | multi-turn tool use loops |

### what's the "aha" moment where the value clicks?

the "aha" moment: when a developer realizes they can build an agent loop where the brain decides what tools to use, executes them, feeds results back, and gets a final answer — all with the same cheap chutes pricing.

```ts
// brain requests tool
const result1 = await atom.ask({
  prompt: 'what is the weather in austin?',
  plugs: { tools: [weatherTool] },
  schema: { output: outputSchema },
});
// result1.calls.tools = [{ slug: 'weather.lookup', input: { city: 'austin' } }]

// developer executes tool, feeds result back
const result2 = await atom.ask({
  on: { episode: result1.episode },
  prompt: [{
    exid: result1.calls.tools[0].exid,
    slug: 'weather.lookup',
    input: { city: 'austin' },
    signal: 'success',
    output: { temp: 75, conditions: 'sunny' },
    metrics: { cost: { time: { milliseconds: 50 } } },
  }],
  plugs: { tools: [weatherTool] },
  schema: { output: outputSchema },
});
// result2.output.content = "It's 75°F and sunny in Austin!"
```

## user experience

### what usecases do folks fulfill? what goals?

| usecase | goal |
|---------|------|
| agentic workflows | build agents that decide which tools to use |
| external data access | fetch real-time data (weather, stocks, apis) |
| computation offload | delegate math/logic to reliable tools |
| database queries | let brain formulate queries, execute externally |
| multi-step tasks | chain tool calls for complex workflows |

### what contract inputs & outputs do they leverage?

**inputs:**
```ts
atom.ask({
  plugs: { tools: BrainPlugToolDefinition[] },  // new
  prompt: string | BrainPlugToolExecution[],    // extended
  // ... extant inputs
})
```

**outputs:**
```ts
BrainOutput<TOutput, 'atom', TPlugs> {
  output: TOutput | null,           // null when tools called
  calls: { tools: BrainPlugToolInvocation[] } | null,  // new
  // ... extant outputs
}
```

### what would it look like to leverage them?

```ts
import { genBrainAtom } from 'rhachet-brains-chutes';
import { z } from 'zod';

// define a tool
const searchTool = {
  slug: 'web.search',
  name: 'Web Search',
  description: 'Search the web for information',
  schema: {
    input: z.object({ query: z.string() }),
    output: z.object({ results: z.array(z.string()) }),
  },
};

// use with atom
const atom = genBrainAtom({ slug: 'chutes/qwen3/coder-next' });
const result = await atom.ask({
  role: {},
  prompt: 'find the latest news about bittensor',
  plugs: { tools: [searchTool] },
  schema: { output: z.object({ summary: z.string() }) },
});

if (result.calls?.tools) {
  // brain wants to use a tool
  console.log('tool requested:', result.calls.tools[0].slug);
} else {
  // brain answered directly
  console.log('answer:', result.output.summary);
}
```

### what timelines do they go through?

1. **tool definition** (once): define tools with slug, description, zod schemas
2. **plug tools** (per ask): pass tools array in plugs.tools
3. **check response** (per ask): check if calls.tools or output populated
4. **execute tools** (if needed): run tool code externally
5. **feed results** (if needed): call ask again with tool executions as prompt
6. **repeat** until output populated

## mental model

### how would users describe this to a friend?

> "i can give the brain a list of tools it can use. when it needs one, it tells me which tool to run and what to pass it. i run the tool, give it the result, and it continues until it has the final answer."

### what analogies or metaphors fit?

- **brain as delegator**: brain doesn't do the work, it delegates to specialists (tools)
- **tool belt**: brain has a tool belt and picks the right tool for the job
- **assistant with phone**: brain can "call" external services when needed

### what terms would they use vs what terms would we use?

| user term | rhachet term |
|-----------|--------------|
| "tool" | `BrainPlugToolDefinition` |
| "tool call" | `BrainPlugToolInvocation` |
| "tool result" | `BrainPlugToolExecution` |
| "tools parameter" | `plugs.tools` |
| "tool response" | `calls.tools` |

## evaluation

### how well does it solve the goals?

| goal | how well solved |
|------|-----------------|
| agentic workflows | ✅ fully enables external tool invocation |
| external data access | ✅ brain can request any defined tool |
| computation offload | ✅ brain can delegate to calculator, etc. |
| multi-step tasks | ✅ continuation via on.episode supported |
| cost efficiency | ✅ same cheap chutes pricing |

### what are the pros? the cons?

**pros:**
- enables agentic workflows on cheap chutes infrastructure
- follows proven xai pattern — low implementation risk
- openai-compatible api means minimal casting complexity
- matches rhachet's plugs.tools interface exactly

**cons:**
- adds complexity to ask() signature
- output can be null (breaking change for strict consumers)
- requires multi-turn handling for tool use loops
- chutes models may vary in tool use capability

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | pit of success |
|----------|----------------|
| brain doesn't call any tool | `calls` is null, `output` is populated |
| brain calls tool but also has content | `calls` populated, `output` null (tool call takes precedence) |
| tool execution fails | `signal: 'error:constraint'` or `'error:unexpected'` with error object |
| invalid tool result fed back | brain continues, may request different approach |
| empty tools array | behaves as if no tools plugged |

## what is awkward?

### what feels off or forced?

- **prompt overloading**: `prompt` being either `string` or `BrainPlugToolExecution[]` feels like two different concerns crammed into one field
- **null output**: returning `null` output when tools are called requires defensive coding at every call site

### where does the design fight the user's mental model?

- users expect `output` to always have a value — null output breaks this mental model
- the term `plugs.tools` is less intuitive than just `tools`
- `BrainPlugToolExecution` is verbose compared to "tool result"

### what tradeoffs feel uncomfortable?

- **verbosity vs clarity**: rhachet types are explicit but verbose
- **flexibility vs simplicity**: supporting both direct answers and tool calls adds complexity
- **consistency vs breaking change**: adding `calls` field is non-breaking, but null `output` may surprise extant consumers
