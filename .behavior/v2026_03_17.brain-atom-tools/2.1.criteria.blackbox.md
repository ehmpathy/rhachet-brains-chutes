# 2.1.criteria.blackbox

## usecase.1 = tool invocation

```
given a developer with a brain atom and a tool definition
  when the developer asks a question that requires the tool
    then the brain returns a tool invocation request
      sothat the developer knows which tool to execute
    then the brain returns null output
      sothat the developer knows the answer is awaited
    then the brain returns the tool's input parameters
      sothat the developer can execute the tool
```

---

## usecase.2 = tool continuation

```
given a developer who received a tool invocation request
  when the developer executes the tool and feeds the result back
    then the brain processes the tool result
      sothat the brain can continue with context
    then the brain returns a final answer
      sothat the developer receives the output they requested
    then the brain returns populated output
      sothat the developer can use the result
```

---

## usecase.3 = direct answer with tools available

```
given a developer with a brain atom and tools plugged
  when the developer asks a question the brain can answer directly
    then the brain returns populated output
      sothat the developer receives the answer
    then the brain returns null tool calls
      sothat the developer knows no tools were needed
```

---

## usecase.4 = tool error signal

```
given a developer who received a tool invocation request
  when the developer executes the tool but it fails
    then the developer can feed back an error signal
      sothat the brain knows the tool failed
    then the brain receives the error context
      sothat the brain can adapt its approach
    then the brain continues with knowledge of the failure
      sothat the brain can try a different approach or report the issue
```

---

## usecase.5 = multi-turn tool loop

```
given a developer with a brain atom and multiple tools plugged
  when the developer asks a question that requires multiple tool calls
    then the brain can request tools one at a time
      sothat the developer executes them sequentially
    then each tool result feeds into the next request
      sothat the brain builds context across turns
    then the final answer incorporates all tool results
      sothat the developer receives a complete response
```

---

## usecase.6 = episode continuation with tools

```
given a developer with an extant episode (prior conversation)
  when the developer continues with a tool-enabled question
    then the brain retains the prior context
      sothat the conversation history is preserved
    then the brain can request tools within the continued episode
      sothat tool use works across sessions
    then the episode grows to include tool exchanges
      sothat the full history is available for future continuation
```

---

## usecase.7 = metrics with tools

```
given a developer who makes a tool-enabled request
  when the request completes (with or without tool calls)
    then the brain returns metrics for the api call
      sothat the developer can track cost and latency
    then the metrics include token counts
      sothat the developer knows input/output size
    then the metrics include elapsed time
      sothat the developer can monitor performance
```

## exchange boundaries

### inputs

```
given a developer who calls atom.ask()
  when the developer provides tools
    then tools are passed via plugs.tools array
      sothat the brain knows available tools
    then each tool has slug, name, description, schema
      sothat the brain understands tool capabilities
  when the developer provides tool results
    then results are passed via prompt as array
      sothat the brain receives execution outcomes
    then each result has exid, slug, input, output, signal
      sothat the brain can correlate and process results
```

### outputs

```
given the brain returns a response
  when tools were invoked
    then output is null
      sothat the developer knows to execute tools first
    then calls.tools contains invocation array
      sothat the developer knows what to execute
    then each invocation has exid, slug, input
      sothat the developer can execute the tool
  when no tools were invoked
    then output is populated with the answer
      sothat the developer receives the result
    then calls is null
      sothat the developer knows no tools needed
```

### errors

```
given the brain encounters an issue
  when the chutes api fails
    then the error propagates to the developer
      sothat they can handle infrastructure issues
  when the model does not support tools
    then the behavior is undefined
      sothat the developer knows to check model compatibility
  when tool input schema is invalid
    then the brain may produce malformed tool calls
      sothat the developer validates tool schemas carefully
```
