# 3.1.1.research.external.product.claims

## summary

this research classifies claims about tool use for rhachet-brains-chutes. claims derive from external research on openai-compatible function call apis, vllm configuration, and model-specific behaviors.

---

## 1. facts [FACT]

indisputable, immutable truths verified by official documentation.

### fact.1 — openai tools parameter structure

> "functions are defined in the `tools` parameter with this schema: type (always 'function'), name (function identifier), description (when/how to use the function), parameters (json schema for input arguments), strict (enforce strict adherence to schema)" [1]

**[FACT]**: the openai tools parameter requires `type: "function"`, `function.name`, `function.description`, `function.parameters`, and optional `strict` field.

source: [1] openai function call docs | [official] | 2025

---

### fact.2 — tool_calls response format

> "chat completions api returns `tool_calls` array with id (used later to submit the function result) and a function with name and json-encoded arguments" [1]

**[FACT]**: tool_calls response contains `id`, `type: "function"`, `function.name`, `function.arguments` (json-encoded string).

source: [1] openai function call docs | [official] | 2025

---

### fact.3 — tool continuation via role "tool"

> "results are passed via role 'tool' messages: { role: 'tool', tool_call_id: 'call_12345xyz', content: 'result_string' }" [1]

**[FACT]**: tool results return to the model via messages with `role: "tool"` and `tool_call_id` that matches the original call.

source: [1] openai function call docs | [official] | 2025

---

### fact.4 — strict mode requires additionalProperties false

> "additionalproperties must be set to false for each object in the parameters. all fields in properties must be marked as required." [1][2]

**[FACT]**: strict mode requires `additionalProperties: false` and all fields marked as `required`.

source: [1] openai function call docs | [official] | 2025
source: [2] openai community strict mode discussion | [practitioner] | 2025

---

### fact.5 — vllm requires parser config per model

> "to enable tool call in vllm, configure: enable-auto-tool-choice flag (mandatory), tool-call-parser flag (specifies the tool parser)" [3]

**[FACT]**: vllm requires `enable_auto_tool_choice: true` and model-specific `tool_call_parser` value.

| model | parser |
|-------|--------|
| mistral | `mistral` |
| llama 3 | `llama3_json` |
| qwen3 | `qwen3_xml` |
| deepseek v3 | `deepseek_v3` |
| hermes | `hermes` |

source: [3] vllm tool call docs | [official] | 2025

---

### fact.6 — chutes uses openai-compatible api

> "chutes supports function call through its optimized serve templates (vllm and sglang), which enables structured data extraction, tool execution, agentic workflows, and external integrations." [4]

**[FACT]**: chutes.ai provides openai-compatible api with function call support via vllm backend.

source: [4] chutes agents-and-tools guide | [official] | 2025

---

### fact.7 — parallel tool calls supported

> "deepseek supports parallel tool call — it can recommend to call multiple tools at once" [5]

> "the parallel_tool_calls parameter manages concurrent execution: true (default, model decides), false (forces sequential, single tool call)" [6]

**[FACT]**: multiple providers support parallel tool calls where the model requests several tools in one response.

source: [5] deepseek function call docs | [official] | 2025
source: [6] mistral function call docs | [official] | 2025

---

### fact.8 — tool_choice controls behavior

> "control how models use available functions: auto (default) — call zero, one, or multiple functions. required — call one or more functions. none — disable function call." [1]

**[FACT]**: tool_choice parameter accepts `"auto"`, `"required"`, `"none"`, or specific function object.

source: [1] openai function call docs | [official] | 2025

---

## 2. assumptions [SUMP]

assumptions that implementers make, either explicitly or implicitly.

### sump.1 — chutes models have tool capability

> "models with excellent function call capabilities include: mistral (7b instruct v0.3), llama 3, qwen" [4]

**[SUMP]**: assumed that qwen3-coder-next, deepseek-v3, and other chutes models support tool use. requires verification per model.

source: [4] chutes agents-and-tools guide | [official] | 2025

---

### sump.2 — tool_calls format identical to openai

**[SUMP]**: assumed that chutes returns `tool_calls` in exact openai format. the openai sdk types provide compile-time validation, but runtime format may vary.

source: pattern inference from [4] and sdk usage in rhachet-brains-xai [7]

---

### sump.3 — json_schema response_format works with tools

**[SUMP]**: assumed that `response_format: { type: 'json_schema' }` can coexist with tools parameter. some providers may conflict.

source: pattern inference from [1] and extant genBrainAtom.ts implementation

---

### sump.4 — vllm parser config active on chutes endpoints

**[SUMP]**: assumed that chutes has vllm configured with appropriate `tool_call_parser` for each model family. not all models may have correct parser config.

source: [3] vllm tool call docs | [official] | 2025
source: [4] chutes agents-and-tools guide | [official] | 2025

---

## 3. open questions [KHUE]

questions that we should investigate or track.

### khue.1 — which chutes models reliably support tools?

> "when the model does not support tools, the behavior is undefined" [vision]

**[KHUE]**: which specific chutes models (qwen3-coder-next, deepseek-v3, kimi-k2, glm-5, hermes-4) have reliable tool use? requires empirical test.

---

### khue.2 — does chutes return tool_calls in exact openai format?

**[KHUE]**: does chutes return `tool_calls` with identical structure? edge cases: empty tool_calls, malformed arguments, extra fields.

---

### khue.3 — should response_format be omitted when tools present?

**[KHUE]**: some providers conflict when both `response_format` and `tools` are specified. should chutes adapter omit json_schema when tools are plugged?

---

### khue.4 — maximum turn limit for tool loops?

> "to avoid the scenario where a program enters an endless loop, a circuit breaker is critical since openai charges you every time you use one of their models" [8]

**[KHUE]**: what maximum turn limit should the circuit breaker use? 10? 20? configurable?

source: [8] openai tool call best practices | [practitioner] | 2025

---

### khue.5 — error signal format for tool failures?

> "instead of the output of the tool call to the model, provide the error reported by the tool to the model and it will figure out what to do" [8]

**[KHUE]**: what exact format should `signal: 'error:constraint'` and `signal: 'error:unexpected'` use for error context?

source: [8] openai tool call best practices | [practitioner] | 2025

## 4. opinions [OPIN]

subjective declarations worth consideration.

### opin.1 — hermes-style recommended for qwen3

> "hermes-style tool use is recommended for qwen3 to maximize function call performance" [9]

**[OPIN]**: qwen documentation recommends hermes-style for tool use, but vllm uses `qwen3_xml` parser. may affect performance.

source: [9] qwen function call docs | [official] | 2025

---

### opin.2 — detailed tool descriptions improve model comprehension

> "use explicit system prompts that define agent capabilities and constraints. provide detailed tool descriptions for better model comprehension." [4]

**[OPIN]**: more detailed tool descriptions lead to better model tool selection.

source: [4] chutes agents-and-tools guide | [official] | 2025

---

### opin.3 — execute parallel tool calls via asyncio

> "execute parallel tool calls via asyncio.gather before response" [4]

**[OPIN]**: developers should execute parallel tool calls concurrently for performance.

source: [4] chutes agents-and-tools guide | [official] | 2025

---

### opin.4 — return errors to model for self-correction

> "return error messages as tool responses for self-correction" [4]

**[OPIN]**: feed tool errors back to the model rather than crash, let model adapt.

source: [4] chutes agents-and-tools guide | [official] | 2025

---

### opin.5 — assume multiple tool calls per response

> "since model responses can include zero, one, or multiple calls, it is best practice to assume there are several" [1][8]

**[OPIN]**: always handle tool_calls as an array, even if typically one.

source: [1] openai function call docs | [official] | 2025
source: [8] openai tool call best practices | [practitioner] | 2025

## 5. convergence signals

when multiple independent sources agree, confidence increases.

### convergence.1 — openai-compatible api format (strong)

**sources [1], [4], [5], [6] agree**: all major llm providers use the same `tools` parameter structure with `type: "function"`, `function.name`, `function.description`, `function.parameters`.

---

### convergence.2 — tool_calls response format (strong)

**sources [1], [5], [6] agree**: tool calls return as array with `id`, `type: "function"`, `function.name`, `function.arguments` (json-encoded string).

---

### convergence.3 — tool continuation via role "tool" (strong)

**sources [1], [5], [6] agree**: tool results feed back via messages with `role: "tool"` and `tool_call_id`.

---

### convergence.4 — strict mode requirements (strong)

**sources [1], [2], [5] agree**: strict mode requires `additionalProperties: false` and all fields marked as `required`.

---

## 6. conflict detection

where sources disagree.

### conflict.1 — llama 3 format variations

**sources [3] and [10] note**: different llama versions use different output formats — llama 3.1/3.3 use json, llama 3.2 1b/3b use pythonic format. vllm normalizes this via `llama3_json` parser.

source: [3] vllm tool call docs | [official] | 2025
source: [10] llama 3.3 model cards | [official] | 2025

---

### conflict.2 — tool_choice terminology

**sources [1] and [6] conflict on terminology**: openai uses `"required"` while mistral uses `"any"` — both mean "must call at least one tool". the openai sdk normalizes this.

source: [1] openai function call docs | [official] | 2025
source: [6] mistral function call docs | [official] | 2025

---

## 7. anti-patterns

what NOT to do, derived from sources.

### antipattern.1 — no additionalProperties in strict mode

> "without strict mode the model 'tries its best' to follow the provided json schema" [2]

**sources [1], [2] warn against**: not set `additionalProperties: false` when strict mode is enabled — the api will reject the schema.

---

### antipattern.2 — no circuit breaker for tool loops

> "to avoid the scenario where a program enters an endless loop, a circuit breaker is critical" [8]

**sources [4], [8] warn against**: no maximum turn limit for tool use loops.

---

### antipattern.3 — swallow tool errors

> "instead of the output of the tool call to the model, provide the error reported by the tool" [8]

**sources [4], [8] warn against**: swallow tool execution errors instead of feed them back to the model.

---

### antipattern.4 — assume all models support tools

> "when the model does not support tools, the behavior is undefined" [vision]

**sources [3], [4] warn against**: assume all models support tool call — each model family requires specific vllm parser config.
