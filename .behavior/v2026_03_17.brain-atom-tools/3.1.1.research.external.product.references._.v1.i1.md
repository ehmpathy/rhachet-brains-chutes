# 3.1.1.research.external.product.references

## summary

this document provides reference materials that ground implementation of tool use for rhachet-brains-chutes. it establishes terms, concepts, demos, and patterns from authoritative sources.

---

## 1. foundational concepts

### concept.1 — function call vs tool call

> "function call refers to the process by which an llm detects that a user request requires external data or action and then produces a structured output (typically in json) that specifies which function to call along with the necessary arguments." [1]

**lesson**: "function call" and "tool call" are synonymous. the openai api uses "tools" parameter but the capability is "function call."

source: [1] deepseek function call docs | [official] | 2025

---

### concept.2 — the tool use loop

> "the typical flow: 1) user sends prompt with tool definitions, 2) model returns tool_calls if needed, 3) developer executes tools, 4) developer feeds results back, 5) model generates final response" [2]

**lesson**: tool use follows a multi-turn loop: request → invoke → execute → feed back → respond.

source: [2] openai function call docs | [official] | 2025

---

### concept.3 — structured outputs via tools

> "function call allows the model to call external tools to enhance its capabilities... produces a structured output (typically in json)" [1]

**lesson**: tools provide structured output guarantee — the model must produce valid json that matches the schema.

source: [1] deepseek function call docs | [official] | 2025

---

### concept.4 — tool_calls array semantics

> "since model responses can include zero, one, or multiple calls, it is best practice to assume there are several" [2][3]

**lesson**: always handle `tool_calls` as an array. the model may request multiple tools in parallel or request none.

source: [2] openai function call docs | [official] | 2025
source: [3] openai tool call best practices | [practitioner] | 2025

---

### concept.5 — idempotent tool execution

> "function call id (used later to submit the function result)" [2]

**lesson**: each tool call has a unique `id` that correlates the invocation with its result. this enables idempotent replay and result correlation.

source: [2] openai function call docs | [official] | 2025

---

## 2. api contract terms

### term.1 — tools parameter

> "functions are defined in the `tools` parameter with this schema: type (always 'function'), name, description, parameters, strict" [2]

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "retrieves current weather for the given location",
    "parameters": {
      "type": "object",
      "properties": {
        "location": { "type": "string" }
      },
      "required": ["location"],
      "additionalProperties": false
    }
  },
  "strict": true
}
```

source: [2] openai function call docs | [official] | 2025

---

### term.2 — tool_calls response

> "chat completions api returns `tool_calls` array with id and a function with name and json-encoded arguments" [2]

```json
{
  "id": "call_12345xyz",
  "type": "function",
  "function": {
    "name": "get_weather",
    "arguments": "{\"location\":\"paris, france\"}"
  }
}
```

source: [2] openai function call docs | [official] | 2025

---

### term.3 — tool role message

> "results are passed via role 'tool' messages: { role: 'tool', tool_call_id: 'call_12345xyz', content: 'result_string' }" [2]

```json
{
  "role": "tool",
  "tool_call_id": "call_12345xyz",
  "content": "{\"temperature\": 72, \"conditions\": \"sunny\"}"
}
```

source: [2] openai function call docs | [official] | 2025

---

### term.4 — tool_choice options

> "control how models use available functions: auto (default), required, none" [2]

| option | behavior |
|--------|----------|
| `"auto"` | model decides whether to use tools |
| `"required"` | model must call at least one tool |
| `"none"` | disable tool call |
| `{ "type": "function", "function": { "name": "..." } }` | force specific function |

source: [2] openai function call docs | [official] | 2025

---

### term.5 — strict mode

> "set strict to true to ensure function calls reliably adhere to the function schema, instead of best effort. under the hood, strict mode leverages structured outputs feature" [4]

**requirements for strict mode:**
- `additionalProperties: false` on all objects
- all fields marked as `required`
- optional fields via `type: ['string', 'null']`

source: [4] openai community strict mode | [practitioner] | 2025

## 5. implementation patterns

### pattern.1 — cast tool definitions

> "the xai adapter provides the pattern to follow" [7]

| cast function | purpose |
|---------------|---------|
| `castIntoXaiToolDef` | converts BrainPlugToolDefinition to openai format |
| `castFromXaiToolCall` | transforms openai tool_calls into BrainPlugToolInvocation |
| `castIntoXaiToolMessages` | converts BrainPlugToolExecution arrays to openai messages |

source: [7] rhachet-brains-xai pr #15 | [practitioner] | 2026-03-17

---

### pattern.2 — zod to json schema

> "zodToJsonSchema from zod-to-json-schema converts zod schemas to valid json schema for the tools parameter" [7]

```ts
import { zodToJsonSchema } from 'zod-to-json-schema';

const jsonSchema = zodToJsonSchema(tool.schema.input, {
  $refStrategy: 'none',
  target: 'openApi3',
});
```

source: [7] rhachet-brains-xai pr #15 | [practitioner] | 2026-03-17

---

### pattern.3 — error signals to model

> "tool execution can fail for various reasons... provide the error reported by the tool to the model and it will figure out what to do" [3]

```ts
// success signal
{ role: 'tool', tool_call_id: id, content: JSON.stringify(result) }

// error signal
{ role: 'tool', tool_call_id: id, content: JSON.stringify({ error: 'rate limit exceeded' }) }
```

source: [3] openai tool call best practices | [practitioner] | 2025

---

### pattern.4 — circuit breaker for loops

> "a circuit breaker is critical since openai charges you every time you use one of their models" [3]

```ts
const MAX_TOOL_TURNS = 10;
let turn = 0;

while (result.calls?.tools && turn < MAX_TOOL_TURNS) {
  // execute tools, feed results
  turn++;
}

if (turn >= MAX_TOOL_TURNS) {
  throw new Error('tool loop exceeded maximum turns');
}
```

source: [3] openai tool call best practices | [practitioner] | 2025

## 7. convergence signals

### convergence.1 — openai-compatible format universal

**sources [1], [2], [6], [9] agree**: the openai tools parameter structure (`type: "function"`, `function.name`, `function.description`, `function.parameters`) is the universal standard adopted by all major providers.

---

### convergence.2 — strict mode requirements universal

**sources [2], [4], [1] agree**: strict mode requires `additionalProperties: false` and all fields as `required` across all providers.

---

### convergence.3 — tool results via role "tool"

**sources [1], [2], [9] agree**: tool results feed back via `role: "tool"` messages with `tool_call_id` correlation.

---

## 8. conflict detection

### conflict.1 — tool_choice terminology varies

**sources [2] and [9] conflict**: openai uses `"required"` while mistral uses `"any"` for "must call at least one tool." the openai sdk normalizes this.

---

### conflict.2 — llama format varies by version

**sources [5] and [10] conflict**: llama 3.1/3.3 use json format, llama 3.2 small models use pythonic format. vllm `llama3_json` parser normalizes this.

---

## 9. anti-patterns to avoid

### antipattern.1 — no strict mode without additionalProperties

**sources [2], [4] warn**: strict mode will reject schemas without `additionalProperties: false`.

---

### antipattern.2 — unbounded tool loops

**sources [3], [6] warn**: tool loops without circuit breaker can loop infinitely and accumulate unbounded costs.

---

### antipattern.3 — swallow tool errors

**sources [3], [6] warn**: swallow tool errors instead of feed them to model prevents self-correction.

---

### antipattern.4 — assume single tool call

**sources [2], [3] warn**: always handle `tool_calls` as array — models can request multiple tools per response.
