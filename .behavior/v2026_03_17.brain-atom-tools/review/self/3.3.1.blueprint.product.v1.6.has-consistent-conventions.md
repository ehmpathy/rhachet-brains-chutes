### our cast function name patterns

```ts
// src/infra/cast/castIntoChutesToolDef.ts line 9
export const castIntoChutesToolDef = (input: {
  tool: BrainPlugToolDefinition;
}): OpenAI.ChatCompletionTool => ({ ... });
// pattern: cast + Into + Provider + Type

// src/infra/cast/castFromChutesToolCall.ts line 8
export const castFromChutesToolCall = (input: {
  call: OpenAI.ChatCompletionMessageToolCall;
}): BrainPlugToolInvocation => ({ ... });
// pattern: cast + From + Provider + Type

// src/infra/cast/castIntoChutesToolMessages.ts line 8
export const castIntoChutesToolMessages = (input: {
  executions: BrainPlugToolExecution[];
}): OpenAI.ChatCompletionMessageParam[] => { ... };
// pattern: cast + Into + Provider + Type
```

**why this is consistent:**

| direction | rhachet pattern | our pattern | match |
|-----------|----------------|-------------|-------|
| input → target | `castInto*` | `castIntoChutesToolDef` | yes |
| input → target | `castInto*` | `castIntoChutesToolMessages` | yes |
| source → output | `castFrom*` | `castFromChutesToolCall` | yes |

the "Chutes" prefix clarifies that these are provider-specific casts. this matches the brain adapter boundary — rhachet provides generic utilities, brain adapters provide provider-specific casts.

**concrete evidence in genBrainAtom.ts:**

### file structure

```
src/
  infra/
    cast/           # new directory for cast functions
      castIntoChutesToolDef.ts
      castFromChutesToolCall.ts
      castIntoChutesToolMessages.ts
  domain.operations/
    atom/
      genBrainAtom.ts  # modified
```

**why this is consistent:**

| structure | rhachet pattern | our pattern | match |
|-----------|----------------|-------------|-------|
| cast location | `domain.operations/*/cast*.ts` or standalone | `infra/cast/` | reasonable |
| one function per file | yes | yes | yes |
| filename = export name | yes | yes | yes |

note: rhachet places some casts in domain.operations, others inline. we chose `infra/cast/` to group provider-specific casts together. this is a reasonable variation since these casts are infrastructure-level (api format conversion), not domain logic.

## conclusion

**no convention divergence found.**

| aspect | extant convention | our implementation | match |
|--------|------------------|-------------------|-------|
| cast names | `castInto*/castFrom*/castTo*` | `castIntoChutesToolDef`, etc. | yes |
| domain op names | `gen*/get*/set*` | no new ops | n/a |
| type names | `Brain*` prefix | use rhachet types | yes |
| file names | `functionName.ts` | yes | yes |
| one export per file | yes | yes | yes |

the "Chutes" prefix in cast function names is intentional — it clarifies these are provider-specific casts for the chutes/openai api format, not generic utilities.
