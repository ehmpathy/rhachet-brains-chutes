### castIntoChutesToolMessages.ts

**blueprint specifies** (lines 60-71):
- input: `{ executions: BrainPlugToolExecution[] }`
- output: `OpenAI.ChatCompletionMessageParam[]`
- builds: assistant message with tool_calls array + tool result messages

**actual implementation** (lines 8-37):
```ts
export const castIntoChutesToolMessages = (input: {
  executions: BrainPlugToolExecution[];
}): OpenAI.ChatCompletionMessageParam[] => {
  // assistant message with tool_calls array (lines 14-25)
  messages.push({
    role: 'assistant',
    content: null,
    tool_calls: input.executions.map((exec) => ({
      id: exec.exid,
      type: 'function',
      function: { name: exec.slug, arguments: JSON.stringify(exec.input) },
    })),
  });

  // tool result messages (lines 28-34)
  for (const execution of input.executions) {
    messages.push({
      role: 'tool',
      tool_call_id: execution.exid,
      content: JSON.stringify(execution.output),
    });
  }
  return messages;
};
```

**holds**: implementation matches blueprint exactly.

### usecase.3 = direct answer with tools available

**criteria**: brain returns populated output, returns null tool calls

**blueprint map**: test coverage table — `[case7] direct answer with tools | usecase.3`

**DEVIATION FOUND**: the blueprint references case7 for usecase.3, but case7 was removed in implementation due to chutes api mutual exclusivity between `response_format` and `tools`. this deviation is intentional and documented in the has-questioned-assumptions review.

**resolution**: the deviation is acceptable. the criteria behavior (return of populated output when brain answers directly) is NOT testable on chutes because without `response_format`, the model may return plain text instead of json. the code handles this case — if no tool_calls, output is parsed as before — but we cannot reliably test it.
