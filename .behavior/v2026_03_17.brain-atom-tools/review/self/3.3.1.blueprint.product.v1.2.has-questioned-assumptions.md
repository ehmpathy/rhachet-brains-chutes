# has-questioned-assumptions

## review date
2026-03-17

## assumptions evaluated

### 1. chutes api supports tools parameter

**stated assumption**: chutes openai-compatible api accepts `tools` parameter

**why it holds**: chutes uses vllm backend which implements openai-compatible function call api. the documentation at chutes.ai confirms openai compatibility. integration tests prove it works: case5 receives tool invocations, case6 processes continuations, case8 measures metrics, case9 handles errors. the cast functions (`castIntoChutesToolDef`, `castFromChutesToolCall`) successfully convert between formats without modification, which confirms format compatibility.

---

### 2. tool_calls format matches openai

**stated assumption**: chutes returns `tool_calls` in openai-compatible format

**why it holds**: `castFromChutesToolCall` extracts three fields: `call.id` → `exid`, `call.function.name` → `slug`, `call.function.arguments` → `input` (via JSON.parse). all fields are present and parseable. the function is 7 lines with no special handling, which means the format is identical. if the format differed, the cast would require conditional logic or fail.

---

### 3. response_format and tools are mutually exclusive

**stated assumption**: can use both `response_format: json_schema` and `tools` together

**issue found**: this assumption was wrong. when both are present, the model prioritizes `response_format` and ignores tools entirely — it answers directly without tool invocation.

**how it was fixed**: added conditional logic to omit `response_format` when tools are present:
```ts
const shouldUseResponseFormat = !hasTools || isToolContinuation;
```

this means: use `response_format` only when (a) no tools plugged, or (b) on tool continuation where we want structured final output.

**lesson learned**: openai-compatible does not mean identical behavior. vllm/chutes treats these parameters as mutually exclusive in practice. always test parameter combinations empirically.

---

### 4. all chutes models support tool use

**stated assumption**: qwen3, deepseek, kimi, glm, hermes all support function call api

**partial validation**: qwen3-coder-next confirmed via tests. other models not tested individually.

**why this is acceptable**: the integration tests use qwen3-coder-next, which is the primary recommended model. other models may vary, but that's a consumer's responsibility to verify model compatibility. the adapter doesn't claim universal tool support — it provides the mechanism.

**could a simpler approach work?** we could have restricted tool use to a subset of models, but this would require a model capability registry that doesn't exist. the simpler approach is: provide the capability, let consumers test their model choices.

---

### 5. tool continuation via prompt array

**stated assumption**: can detect tool continuation via prompt type check

**why it holds**: `askInput.prompt` is typed as `string | BrainPlugToolExecution[]`. these are mutually exclusive types — a string is never an array. `Array.isArray()` is the idiomatic way to discriminate union types in typescript. the xai implementation uses the same pattern. case6 proves it works: first call returns tool invocation, second call with array prompt returns final output.

**what if the opposite were true?** if prompt could be both string and tool executions simultaneously, we'd need a wrapper type. but the rhachet api design explicitly separates these: initial prompt is string, continuation prompt is tool results. the type system enforces this.

---

### 6. json.parse on tool arguments

**stated assumption**: tool call arguments are always valid json

**why it holds for tested models**: qwen3-coder-next returns well-formed json arguments in all test cases. the model is trained on function call patterns and produces valid json.

**risk acknowledged**: other models or edge cases might produce malformed json. if `JSON.parse` throws, the error propagates to the consumer.

**why this is acceptable**: per criteria, errors propagate to developers. the consumer can catch parse errors if needed. adding try/catch with custom error would add complexity for an edge case that hasn't manifested.

---

## issues found and fixed

### issue: response_format + tools mutual exclusivity

**what we assumed without evidence**: that openai-compatible means both parameters work together

**what the opposite showed**: model ignores tools when response_format is present

**how fixed**: conditional `shouldUseResponseFormat` based on `hasTools` and `isToolContinuation`

---

### issue: case7 "direct answer with tools" is unreliable

**what we assumed**: brain can answer directly without tool use even when tools are available

**evidence against**: without `response_format`, model may return plain text instead of json. this breaks schema parse.

**resolution**: removed case7 from test suite. documented in `has-questioned-deletables` review. the feature is inherently unreliable with chutes api due to the mutual exclusivity constraint.

---

## conclusion

six assumptions evaluated. two required correction. the critical discovery — response_format and tools mutual exclusivity — led to implementation changes and test scope reduction. the final implementation handles this correctly by conditionally including response_format only when appropriate.
