# has-pruned-backcompat

## review date
2026-03-17

## backwards compatibility concerns evaluated

### 1. output type change (TOutput | null)

**what changed**: before tools, `output` was always `TOutput`. now `output` can be `null` when tools are called.

**was this explicitly requested?**

yes — criteria 2.1 usecase.1 states:
> "then the brain returns null output sothat the developer knows the answer is awaited"

this is not implied backwards compat — it's the specified behavior.

**evidence this holds — code path analysis:**

```ts
// genBrainAtom.ts line ~231
if (toolCalls?.length) {
  return new BrainOutput({
    output: null,  // explicitly null when tools called
    calls: { tools: toolInvocations },
    // ...
  });
}
// else clause at line ~248
return new BrainOutput({
  output,  // populated when no tools called
  calls: null,
  // ...
});
```

the code path guarantees: tools called → output null; no tools → output populated. no "safe" intermediate state added.

**why extant consumers are unaffected:**

extant calls don't pass `plugs.tools`. without tools, the code path never reaches the `if (toolCalls?.length)` branch. output is always `TOutput` for tool-less calls.

### 2. calls field addition

**what changed**: `BrainOutput` now has `calls: { tools: BrainPlugToolInvocation[] } | null` field.

**was this explicitly requested?**

yes — criteria 2.1 usecase.1 states:
> "then calls.tools contains invocation array sothat the developer knows what to execute"

**evidence this holds — type analysis:**

the field type `{ tools: ... } | null` means extant consumers who don't access `calls` are unaffected. typescript doesn't require destructure of all fields. this is an additive change that does not break extant code.

**concrete proof:**

```ts
// extant consumer code (unchanged)
const result = await atom.ask({ prompt, schema });
console.log(result.output);  // works — output still typed TOutput
console.log(result.episode); // works — episode still typed BrainEpisode

// new field is simply ignored if not accessed
// no runtime error, no type error
```

### 3. prompt type extension

**what changed**: `prompt` can now be `string | BrainPlugToolExecution[]`.

**was this explicitly requested?**

yes — criteria 2.1 usecase.2 states:
> "results are passed via prompt as array sothat the brain receives execution outcomes"

**evidence this holds — type theory:**

union type extension is backwards compatible by definition. if a function accepted `string` and now accepts `string | X`, all extant calls with `string` remain valid.

**concrete proof:**

```ts
// extant code (unchanged)
await atom.ask({ prompt: 'hello world', schema });  // string → valid

// new code (added capability)
await atom.ask({ prompt: toolExecutions, schema });  // array → valid
```

no migration path needed because no extant code breaks.

---

### 4. TPlugs generic addition

**what changed**: `ask` signature now has `TPlugs extends BrainPlugs = BrainPlugs` generic.

**was this explicitly requested?**

yes — required by rhachet's `BrainOutput<TOutput, 'atom', TPlugs>` type signature. we did not choose to add this generic — the parent library mandates it for tool-aware return types.

**evidence this holds — typescript inference:**

```ts
// extant code — generic inferred as default
const result = await atom.ask({ prompt, schema });
// TPlugs inferred as BrainPlugs (default)

// new code — generic explicit for tool type safety
const result = await atom.ask<OutputType, typeof plugs>({
  prompt,
  plugs,
  schema,
});
```

default value `= BrainPlugs` ensures extant calls compile without modification.

## backwards compat we did NOT add

### 1. no deprecation log for non-tool usage

**was this explicitly requested?** no — never mentioned in wish, vision, or criteria.

**did we add it "to be safe"?** no — no log statement for non-tool calls exists.

**why it would be wrong**: deprecation implies future removal. tool use is additive, not a replacement path.

### 2. no fallback for null output

**was this explicitly requested?** no — criteria says output should be null, not "null with fallback".

**did we add it "to be safe"?** no — the code returns `output: null` directly.

**why it would be wrong**: a fallback (empty object, thrown error) would violate criteria and confuse consumers about when tools were called.

### 3. no migration helper

**was this explicitly requested?** no — wish says "same tool use extension", not "backwards compat layer".

**did we add it "to be safe"?** no — no `askLegacy`, `askV1`, or wrapper exists.

**why it would be wrong**: creates maintenance burden. extant behavior is preserved — no migration needed.

### 4. no version flag

**was this explicitly requested?** no — never mentioned.

**did we add it "to be safe"?** no — response has no `_version` or `_toolsEnabled` field.

**why it would be wrong**: version detection should happen via types, not runtime flags. typescript already tells consumers the shape.

---

## conclusion

four backwards compat changes identified — all explicitly required by criteria or type constraints. four potential "safety" additions considered — all correctly omitted. no backwards compat was added speculatively. extant consumers who don't use tools experience zero behavioral or type change.
