# 1.vision.2.has-questioned-assumptions

## assumptions surfaced and questioned

### assumption 1: chutes api supports the `tools` parameter

**what do we assume here without evidence?**
- chutes.ai advertises "openai-compatible api"
- we assume this includes the `tools` parameter

**what evidence supports this assumption?**
- chutes docs claim openai compatibility
- chutes uses openai sdk with baseURL override (same as xai)
- no explicit confirmation in vision

**what if the opposite were true?**
- if chutes doesn't support tools, entire feature is blocked
- would need to find alternative or wait for chutes to add support

**did the wisher actually say this, or did we infer it?**
- wisher said "implement the same tool use"
- wisher may assume chutes supports it (like xai does)
- we inferred support from openai compatibility claim

**verdict: RISK — MUST VERIFY**
- add to research: verify tools parameter in chutes api
- this is a blocker assumption

### assumption 7: chutes latency is acceptable for tool use loops

**what do we assume here without evidence?**
- tool use requires multiple api round trips
- chutes can handle these round trips fast enough to be useful

**what evidence supports this assumption?**
- extant tests use 120s timeout — suggests chutes can be slow
- briefs mention chutes is decentralized with variable latency

**what if the opposite were true?**
- multi-turn tool use could take minutes per loop
- users may abandon chutes for tool use in favor of faster providers
- we may need to document latency expectations

**counterexamples:**
- ref.chutes-performance-stability-capacity.md notes "occasional throttle in heavy global traffic periods"
- miner-dependent availability could cause failures mid-loop

**verdict: RISK — SHOULD DOCUMENT**
- latency is a known chutes trade-off
- not a blocker, but should be documented as caveat
