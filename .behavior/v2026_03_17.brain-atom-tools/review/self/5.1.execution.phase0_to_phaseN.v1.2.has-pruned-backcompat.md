# has-pruned-backcompat

## review date
2026-03-17

## methodology

reviewed the implementation for backwards compatibility concerns that were not explicitly requested. for each backcompat concern, asked:
1. did the wisher explicitly say to maintain this compatibility?
2. is there evidence this backwards compat is needed?
3. or did we assume it "to be safe"?

---

## backwards compatibility analysis

### api signature changes

**change**: `ask()` method now accepts `TPlugs` generic parameter

| question | answer |
|----------|--------|
| explicit request? | no — wisher requested tool use, not API stability |
| evidence needed? | yes — consumers rely on ask() signature |
| assumed to be safe? | partially |

**analysis**: the TPlugs generic is additive with a default value (`BrainPlugs = BrainPlugs`). callers who don't use tools see no change. this is a natural consequence of the feature, not explicit backcompat.

**verdict**: holds — additive change with defaults, not a backcompat shim

---

### prompt type union

**change**: `prompt` type extended from `string` to `string | BrainPlugToolExecution[]`

| question | answer |
|----------|--------|
| explicit request? | yes — vision line 82: "prompt can be tool execution results" |
| evidence needed? | yes — all consumers pass strings |
| assumed to be safe? | no — vision explicitly requested |

**analysis**: union type is backwards compatible. `string` callers work unchanged. `BrainPlugToolExecution[]` callers use the new feature.

**verdict**: holds — explicit requirement in vision

---

### output nullability

**change**: `output` can now be `null` when tools are called

| question | answer |
|----------|--------|
| explicit request? | yes — vision line 26: "result.output = null (brain wants to use tool first)" |
| evidence needed? | n/a — new behavior for new feature |
| assumed to be safe? | no — vision explicitly stated |

**analysis**: this is an incompatible change for consumers who assume output is always populated. however, it was explicitly requested in the vision.

**verdict**: holds — explicit requirement, consumers must handle null

---

### calls property addition

**change**: `BrainOutput` now includes `calls: { tools: BrainPlugToolInvocation[] } | null`

| question | answer |
|----------|--------|
| explicit request? | yes — vision line 25: "result.calls.tools = [...]" |
| evidence needed? | n/a — new property |
| assumed to be safe? | n/a — additive |

**analysis**: additive property, does not break consumers who don't access it.

**verdict**: holds — explicit requirement, additive change

---
