# has-no-silent-scope-creep

## review date
2026-03-17

## methodology

examined every file change and codepath to determine if scope creep occurred. for each change, asked:
1. was this requested by the wish?
2. was this in the blueprint?
3. is this a "while we were in there" change?

---

## scope verification

### wish scope

the wish requested (from `.behavior/v2026_03_17.brain-atom-tools/0.wish.md`):
- research how rhachet-brains-xai implemented tool use
- research rhachet's tool use api
- implement the same tool use extension
- implement the same tool use test coverage

**interpretation:** add tool use capability to rhachet-brains-chutes, mirror xai implementation, add test coverage.

### files changed

| file | in wish scope? | justification |
|------|---------------|---------------|
| castIntoChutesToolDef.ts | ✓ | tool use cast function — converts BrainPlugToolDefinition to openai format |
| castFromChutesToolCall.ts | ✓ | tool use cast function — converts openai tool_call to BrainPlugToolInvocation |
| castIntoChutesToolMessages.ts | ✓ | tool use cast function — converts BrainPlugToolExecution[] to openai messages |
| genBrainAtom.ts | ✓ | tool use support in ask() — TPlugs generic, tools param, tool_calls extraction |
| genBrainAtom.integration.test.ts | ✓ | tool use test cases — [case5], [case6], [case8], [case9] |
| package.json | ✓ | rhachet version update for tool use types |
| pnpm-lock.yaml | ✓ | lockfile for package.json |

**why each file is in scope:**

1. **castIntoChutesToolDef.ts** — the wish says "implement the same tool use extension". xai has this cast function (castIntoXaiToolDef.ts), so we need the chutes equivalent.

2. **castFromChutesToolCall.ts** — same logic. xai has castFromXaiToolCall.ts for tool call extraction. chutes needs the same.

3. **castIntoChutesToolMessages.ts** — same logic. xai has castIntoXaiToolMessages.ts for tool continuation. chutes needs the same.

4. **genBrainAtom.ts** — the wish says "implement the same tool use extension". the extension lives in the ask() function. changes include TPlugs generic, prompt union type, tools param, conditional response_format, tool_calls extraction.

5. **genBrainAtom.integration.test.ts** — the wish explicitly says "implement the same tool use test coverage". we added [case5] tool invocation, [case6] tool continuation, [case8] metrics with tools, [case9] tool error signal.

6. **package.json** — rhachet version update to 0.9.0 for BrainAtomPlugs type with tools field. this is required for the implementation.

7. **pnpm-lock.yaml** — lockfile update follows package.json. not a behavioral change.

## additions analysis

**question: did we add implementation details not in the blueprint?**

| addition | in blueprint? | scope creep? |
|----------|--------------|--------------|
| `tool_choice: 'auto'` (line 124) | implicit in "tools parameter" | no — necessary for correct api call |
| `isToolContinuation` check (line 113) | implicit in "conditional response_format" | no — necessary for correct api behavior |
| `hasTools` variable (line 112) | not explicit | no — local variable for readability, not new feature |
| comment at lines 128-129 | not in blueprint | no — documents api limitation, not behavioral change |

**why these are not scope creep:**
- `tool_choice: 'auto'` — openai api requires tool_choice for function call behavior. omission would break tool use.
- `isToolContinuation` — blueprint says "response_format when no tools OR when in continuation mode". implementation checks this.
- `hasTools` variable — local variable for readability. no semantic difference from inlined check.
- comment — documents api limitation discovered in implementation. documentation is not scope creep.
