# 1.vision.1.has-questioned-requirements

## requirements reviewed

### requirement 1: tool use for brains

**who said this was needed?**
- the wisher, explicitly in 0.wish.md
- references xai pr #15 which adds tool use

**what evidence supports this requirement?**
- wish links to tested xai implementation
- rhachet package readme documents `plugs.tools` interface
- chutes.ai uses openai-compatible api which supports tools

**what if we didn't do this?**
- consumers couldn't build agentic workflows with chutes
- would miss feature parity with xai, openai, anthropic suppliers
- would limit usefulness of cheap chutes inference

**verdict: HOLDS** — core requirement, explicitly requested, evidence clear

### requirement 2: follow xai pattern as reference

**who said this was needed?**
- wisher: "research how it did that... implement the same"
- explicit reference to xai pr #15

**what evidence supports this requirement?**
- xai pr #15 is a tested implementation
- xai uses openai-compatible api (same as chutes)
- pattern is proven and reviewed

**could we achieve the goal in a simpler way?**
- could write from scratch, but higher risk
- could follow anthropic pattern, but different api shape
- xai pattern is most similar since both use openai sdk

**verdict: HOLDS** — lower risk, explicit instruction, appropriate match

### requirement 3: same test coverage as xai

- these are the core behaviors needed for tool use

**is the scope too large?**
- 6 new test cases is reasonable
- matches what xai implemented
- covers the essential behaviors

### requirement 4: TPlugs generic on ask()

**who said this was needed?**
- implicit from rhachet's BrainPlugs interface
- xai pattern uses this approach

**what evidence supports this requirement?**
- rhachet exports BrainPlugs, BrainPlugToolDefinition types
- xai pr adds TPlugs generic to ask() signature
- enables type-safe tool definitions

**could we achieve this simpler?**
- could use `any` for plugs, but loses type safety
- could use hardcoded types, but less flexible
- generic matches rhachet's design

**verdict: HOLDS** — matches framework design, enables type safety

---

### requirement 5: prompt as string | BrainPlugToolExecution[]

**who said this was needed?**
- implicit from xai pattern and rhachet interface
- needed to feed tool results back

**what evidence supports this requirement?**
- xai implements prompt overload (union type)
- rhachet's BrainPlugToolExecution is the result type
- openai api requires tool_results in messages

**what feels awkward?**
- two different concerns in one field
- noted in vision "what is awkward" section

**could we do this differently?**
- separate field like `toolResults`? diverges from xai
- callback pattern? more complex
- union type is consistent with xai

**verdict: HOLDS with DISCOMFORT** — awkward but matches pattern, alternative would diverge

---

### requirement 6: output can be null when tools called

**who said this was needed?**
- implicit from openai/xai behavior
- tools and content are mutually exclusive in response

**what evidence supports this requirement?**
- openai chat completions: content null when tool_calls present
- xai implements this behavior
- brain must complete tool calls before answer

**does this break consumers?**
- yes, consumers that expect output to always exist will break
- noted in vision cons section

**could we mitigate?**
- could return empty object, but inconsistent with api
- could use discriminated union, but complicates types
- null is honest about state

**verdict: HOLDS with CAUTION** — breaks extant consumers, but matches api reality
