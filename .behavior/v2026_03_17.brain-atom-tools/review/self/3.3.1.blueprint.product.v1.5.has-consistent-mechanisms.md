## mechanisms evaluated

### 1. castIntoChutesToolDef.ts

**does the codebase already have a mechanism for this?**

no — this is the first tool use implementation in rhachet-brains-chutes.

**do we duplicate extant utilities?**

no — rhachet provides `BrainPlugToolDefinition` domain object but NOT the cast to openai format. the cast is necessarily provider-specific:

```ts
// openai format (what chutes needs)
{
  type: 'function',
  function: { name, description, parameters }
}

// vs anthropic format (what anthropic needs)
{
  name, description, input_schema
}
```

the structure differs — there is no generic "castToolToApi" that works for both. each adapter must provide its own cast.

**could we reuse an extant component?**

yes, partially — we correctly reuse `z.toJSONSchema()` from zod:

```ts
// castIntoChutesToolDef.ts line 16
parameters: z.toJSONSchema(input.tool.schema.input),
```

the rest is field map specific to openai api, which cannot be abstracted without loss of clarity.

### 2. castFromChutesToolCall.ts

**does the codebase already have a mechanism for this?**

no — this is new functionality.

**do we duplicate extant utilities?**

no — rhachet provides `BrainPlugToolInvocation` domain object but NOT the cast from openai format. the extraction logic is specific to openai's response structure:

```ts
// openai tool_call structure
{
  id: "call_abc123",
  type: "function",
  function: {
    name: "calculator.multiply",
    arguments: '{"a":7,"b":8}'  // JSON string, not object
  }
}

// rhachet BrainPlugToolInvocation
{
  exid: "call_abc123",
  slug: "calculator.multiply",
  input: { a: 7, b: 8 }  // parsed object
}
```

the `arguments` field is a JSON string in openai format — we must parse it. this is provider-specific behavior.

**could we reuse an extant component?**

no — `JSON.parse` is standard library. the field map (`id` → `exid`, `function.name` → `slug`, `function.arguments` → `input`) is specific to openai's format.

### 3. castIntoChutesToolMessages.ts

**does the codebase already have a mechanism for this?**

no — this is new functionality.

**do we duplicate extant utilities?**

no — rhachet provides `BrainPlugToolExecution` domain object but NOT the cast to openai message format. the message structure is specific to openai's continuation format:

```ts
// openai requires TWO message types for tool continuation:

// 1. assistant message with tool_calls (lines 14-25)
{
  role: 'assistant',
  content: null,
  tool_calls: [{ id, type: 'function', function: { name, arguments } }]
}

// 2. tool result messages (lines 28-34)
{
  role: 'tool',
  tool_call_id: "call_abc123",
  content: '{"result":56}'
}
```

this two-message structure is dictated by openai's api — anthropic uses a different structure (`tool_result` content blocks). no generic abstraction exists.

**could we reuse an extant component?**

no — `JSON.stringify` is standard library. the message structure is provider-specific.

| mechanism | extant? | duplicates? | why new is correct |
