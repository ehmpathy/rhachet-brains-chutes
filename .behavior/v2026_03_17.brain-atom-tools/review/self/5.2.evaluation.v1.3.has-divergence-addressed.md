### evidence of api limitation

**the code comment (line 128-129):**
```ts
// build response_format when no tools OR when in continuation mode
// (json_schema and tool calls are mutually exclusive in model behavior)
```

**the conditional logic (lines 130-142):**
```ts
const shouldUseResponseFormat = !hasTools || isToolContinuation;
const responseFormatParam = shouldUseResponseFormat
  ? {
      response_format: {
        type: 'json_schema' as const,
        json_schema: {
          name: 'response',
          strict: true,
          schema: jsonSchema,
        },
      },
    }
  : {};
```

**why this proves the limitation:**
- when `hasTools` is true AND `isToolContinuation` is false → `shouldUseResponseFormat` is false
- this means no `response_format: { type: 'json_schema' }` in api call
- without json_schema constraint, model may return prose instead of JSON
- test would need to parse output, but output format is unpredictable

the case is fully handled by the code path. the limitation is only in test coverage.

### stress test: is this laziness or genuine limitation?

**question 1: is this truly an api limitation, or avoidance of work?**

**answer:** genuine api limitation. the openai-compatible api (and vllm backend) does not support `response_format: { type: 'json_schema' }` with `tools` simultaneously. this is documented behavior:
- when both are present, model behavior is undefined
- openai's function call api forces a choice: structured output OR tool use
- chutes inherits this limitation from the openai-compatible layer

**evidence in code:** the comment at line 128-129 explicitly documents this mutual exclusivity. this was discovered by the xai reference implementation and applies here.

**question 2: could we test this differently?**

**answer:** no reliable test possible without json_schema constraint.

options considered:
1. **add a flaky test** — bad practice, violates test reliability principles
2. **mock the api** — violates the "never mock" integration test principle per briefs
3. **use a different assertion** — no reliable assertion possible when output format is unpredictable
4. **test with specific models that always output JSON** — model behavior is not guaranteed; varies by prompt, temperature, model version

**question 3: could this divergence cause problems later?**

**answer:** no. the code handles the case (lines 244-254). users who plug tools and receive direct answers will get valid output. the limitation is purely in test coverage, not behavior.

**evidence:** the code path `JSON.parse(content)` + `schema.output.parse(parsed)` validates output regardless of whether tools were plugged. if the model returns valid JSON, it works. if the model returns prose, `JSON.parse` throws and user gets clear error.

**question 4: is the backup documented properly?**

**answer:** yes. documentation exists in:
1. evaluation artifact `5.2.evaluation.v1.i1.md` divergence resolution table with rationale
2. earlier review `.behavior/v2026_03_17.brain-atom-tools/review/self/3.3.1.blueprint.product.v1.2.has-questioned-assumptions.md` documents this limitation
3. code comment at line 128-129 explains the mutual exclusivity

---

### repair option analysis

**could we repair this instead of backup?**

| option | feasibility | why not |
|--------|-------------|---------|
| add a flaky test | no | violates test reliability principles; intermittent failures waste engineer time |
| mock the api | no | violates "never mock" integration test principle; mocks lie |
| use a model that always returns JSON | no | model behavior varies by prompt and version; no guarantee |
| assert on error instead of success | partial | we could test that error is thrown when model returns prose, but this tests failure path not success |
| skip output validation in test | no | defeats the purpose of verification |

**conclusion:** repair is not feasible. the test would be unreliable by design. backup is the correct resolution.

### verdict

**backup is valid.** the divergence is due to genuine api limitation, not avoidance of work.

evidence summary:

2. code handles the case — output parsed and validated (lines 244-254)
3. test would be unreliable — no json_schema constraint means unpredictable output format
4. backup is documented — in evaluation artifact, earlier review, and code comment
