# 3.1.1.research.external.product.access

## summary

this research documents the remote access required to implement tool use for rhachet-brains-chutes. the implementation follows the openai-compatible function call pattern via chutes.ai's vllm-based inference infrastructure.

---

## 1. remote repositories accessed

### 1.1 chutes.ai inference api

**endpoint:** `https://llm.chutes.ai/v1` (or custom deployed chute endpoints)
**auth:** bearer token with `cpk_` prefix via `CHUTES_API_KEY`
**interface:** openai-compatible rest api

> "chutes supports function call through its optimized serve templates (vllm and sglang), which enables structured data extraction, tool execution, agentic workflows, and external integrations." [1]

> "models with excellent function call capabilities include: mistral (7b instruct v0.3), llama 3, qwen" [1]

### 1.2 rhachet type system

**package:** `rhachet` (peer dependency >= 1.21.4)
**types accessed:**
- `BrainPlugToolDefinition` — tool schema with slug, name, description, zod schemas
- `BrainPlugToolInvocation` — output when brain requests tool execution
- `BrainPlugToolExecution` — input when developer feeds tool results back
- `BrainOutput<TOutput, 'atom', TPlugs>` — response type with `calls.tools` field

## 2. api contracts

### 2.1 openai function call specification

the openai function call api defines the standard contract that chutes implements [2][3][8].

**tools parameter structure:**

> "functions are defined in the `tools` parameter with this schema: type (always 'function'), name (function identifier), description (when/how to use the function), parameters (json schema for input arguments), strict (enforce strict adherence to schema)" [3]

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "retrieves current weather for the given location",
    "parameters": {
      "type": "object",
      "properties": {
        "location": { "type": "string" },
        "units": { "type": "string", "enum": ["celsius", "fahrenheit"] }
      },
      "required": ["location", "units"],
      "additionalProperties": false
    }
  },
  "strict": true
}
```

**tool_choice options:**

> "control how models use available functions: auto (default) — call zero, one, or multiple functions. required — call one or more functions. none — disable function call." [3]

| option | behavior |
|--------|----------|
| `"auto"` | model decides whether to use tools |
| `"required"` | model must call at least one tool |
| `"none"` | disable tool call |
| `{ "type": "function", "function": { "name": "..." } }` | force specific function |

**response format with tool_calls:**

> "chat completions api returns `tool_calls` array with id (used later to submit the function result) and a function with name and json-encoded arguments" [3]

```json
{
  "id": "call_12345xyz",
  "type": "function",
  "function": {
    "name": "get_weather",
    "arguments": "{\"location\":\"paris, france\"}"
  }
}
```

**tool result submission:**

> "results are passed via role 'tool' messages: { role: 'tool', tool_call_id: 'call_12345xyz', content: 'result_string' }" [3]

### 2.2 strict mode requirements

strict mode ensures reliable schema adherence via structured outputs [3][23].

> "set strict to true to ensure function calls reliably adhere to the function schema, instead of best effort. under the hood, strict mode leverages structured outputs feature" [23]

**requirements:**

> "additionalproperties must be set to false for each object in the parameters. all fields in properties must be marked as required. denote optional fields via 'type': ['string', 'null']" [3][23]

```json
{
  "parameters": {
    "type": "object",
    "properties": {
      "required_field": { "type": "string" },
      "optional_field": { "type": ["string", "null"] }
    },
    "required": ["required_field", "optional_field"],
    "additionalProperties": false
  },
  "strict": true
}
```

### 2.3 vllm tool call parser config

chutes uses vllm for model serve, which requires specific config for tool call [4][5].

> "to enable tool call in vllm, configure: enable-auto-tool-choice flag (mandatory), tool-call-parser flag (specifies the tool parser), chat-template flag (optional, defines tool handle)" [4]

**supported parsers by model family:**

| model | parser | source |
|-------|--------|--------|
| mistral | `--tool-call-parser mistral` | [4] |
| llama 3 | `--tool-call-parser llama3_json` | [4] |
| qwen3 | `--tool-call-parser qwen3_xml` | [4] |
| qwen3-coder | `--tool-call-parser qwen3_xml` | [4] |
| deepseek v3 | `--tool-call-parser deepseek_v3` | [4] |
| hermes | `--tool-call-parser hermes` | [4] |

**chutes deployment example:**

> "to enable tool support, deploy with these engine arguments: enable_auto_tool_choice: true, tool_call_parser: 'mistral' (or 'llama3_json')" [1]

```python
engine_args = {
    "enable_auto_tool_choice": True,
    "tool_call_parser": "mistral",
    "max_model_len": 8192
}
```

## 3. model-specific contracts

### 3.1 qwen3 function call

> "qwen3 excels in tool call capabilities, with qwen-agent recommended to make the best use of agentic ability of qwen3, as it encapsulates tool-call templates and tool-call parsers internally" [6]

> "hermes-style tool use is recommended for qwen3 to maximize function call performance" [6]

> "in september 2025, qwen3-vl tool-call demo was added... in july 2025, qwen3-coder tool-call demo was added with native api tool call interface support, such as vllm's built-in tool call parse" [6]

### 3.2 deepseek v3 function call

> "function call allows the model to call external tools to enhance its capabilities. function call refers to the process by which an llm detects that a user request requires external data or action and then produces a structured output (typically in json) that specifies which function to call along with the necessary arguments." [7]

**parallel tool calls:**

> "deepseek supports parallel tool call — it can recommend to call multiple tools at once" [7]

**strict mode:**

> "in strict mode, the model strictly adheres to the format requirements of the function's json schema when it outputs a function call, to ensure the model's output complies with the user's definition" [7]

**openai sdk compatibility:**

> "you can use the openai sdk with deepseek via baseurl set to 'https://api.deepseek.com'. the model used is 'deepseek-chat' for standard requests." [7]

### 3.3 mistral function call

> "tool_choice parameter controls which tool is called by the model. 'none' means the model will not call any tool. 'auto' means the model can pick between message generation or tool call. 'any' or 'required' means the model must call one or more tools." [9]

**parallel tool calls:**

> "the parallel_tool_calls parameter manages concurrent execution: true (default, model decides), false (forces sequential, single tool call)" [9]

**response format:**

> "when a model determines tool use is necessary, it returns a tool call with: function name, arguments as json, tool_call_id for track. the developer then executes the function and returns results via a 'tool' role message" [9]

### 3.4 llama 3 function call

> "for llama 3.1 and 3.3: the meta-llama/llama-3.3-70b-instruct model can be prompted to construct either json or python tool calls in its responses, with its default prompt setup for json tool call" [10]

> "for llama 3.2 1b and 3b instruct models, a new format for zero-shot function call is used, where the assistant responds with function calls in a pythonic format like [func1(params_name=params_value...), func2(params)] instead of json or function tags defined in llama 3.1" [10]

## 4. best practices

### 4.1 tool definition best practices

> "use explicit system prompts that define agent capabilities and constraints. provide detailed tool descriptions for better model comprehension." [1]

> "tools need to describe the type, function name, function description, function parameters, and the required parameter for the function" [9]

### 4.2 error handle best practices

> "tool execution can fail for various reasons, and while the openai documentation doesn't explain this scenario in detail, the answer is straightforward: instead of the output of the tool call to the model, provide the error reported by the tool to the model and it will figure out what to do" [17]

> "return error messages as tool responses for self-correction" [1]

> "for most models, you need to provide some guidelines in the system prompt to help the model handle these situations effectively" [17]

### 4.3 multi-turn loop best practices

> "since model responses can include zero, one, or multiple calls, it is best practice to assume there are several" [3][17]

> "circuit breaker pattern: to avoid the scenario where a program enters an endless loop, a circuit breaker is critical since openai charges you every time you use one of their models" [17]

### 4.4 parallel execution best practices

> "execute parallel tool calls via asyncio.gather before response" [1]

## 5. anti-patterns to avoid

### 5.1 schema validation failures

> "without strict mode the model 'tries its best' to follow the provided json schema" [23]

**sources [3], [23] warn against:** not set `additionalProperties: false` when strict mode is enabled — the api will reject the schema.

### 5.2 endless loops

> "to avoid the scenario where a program enters an endless loop, a circuit breaker is critical" [17]

**sources [1], [17] warn against:** no maximum turn limit for tool use loops.

### 5.3 ignore tool errors

> "instead of the output of the tool call to the model, provide the error reported by the tool" [17]

**sources [1], [17] warn against:** swallow tool execution errors instead of feed them back to the model.

### 5.4 model incompatibility

> "when the model does not support tools, the behavior is undefined" [vision]

**sources [1], [4] warn against:** assume all models support tool call — each model family requires specific vllm parser config.

## 7. convergence signals

### 7.1 openai-compatible api format (strong convergence)

**sources [1], [3], [7], [9] agree:** all major llm providers (openai, deepseek, mistral, chutes) use the same `tools` parameter structure with `type: "function"`, `function.name`, `function.description`, `function.parameters`.

### 7.2 tool_calls response format (strong convergence)

**sources [3], [7], [9] agree:** tool calls are returned as an array with `id`, `type: "function"`, `function.name`, `function.arguments` (json-encoded string).

### 7.3 tool continuation via role "tool" (strong convergence)

**sources [3], [7], [9] agree:** tool results are fed back via messages with `role: "tool"` and `tool_call_id` that matches the original call.

### 7.4 strict mode requirements (strong convergence)

**sources [3], [7], [23] agree:** strict mode requires `additionalProperties: false` and all fields marked as `required`.

---

## 8. conflict detection

### 8.1 llama 3 format variations

**sources [4] and [10] note:** different llama versions use different output formats — llama 3.1/3.3 use json, llama 3.2 1b/3b use pythonic format. vllm normalizes this via `llama3_json` parser.

### 8.2 tool_choice terminology

**sources [3] and [9] conflict on terminology:** openai uses `"required"` while mistral uses `"any"` — both mean "must call at least one tool". the openai sdk normalizes this.

## implementation implications

based on this research, the chutes adapter should:

1. **use openai sdk** with `baseURL: 'https://llm.chutes.ai/v1'` — the api is openai-compatible
2. **cast tool definitions** from `BrainPlugToolDefinition` to openai's `tools` array format
3. **extract tool_calls** from response `choices[0].message.tool_calls`
4. **cast tool results** to openai's `role: "tool"` message format with `tool_call_id`
5. **support parallel tool calls** — the response may contain multiple tool_calls
6. **handle strict mode** — set `additionalProperties: false` and all fields as required
7. **implement circuit breaker** — limit maximum tool use turns to prevent endless loops
