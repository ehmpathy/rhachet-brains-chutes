# research: remote access for rhachet-brains-chutes

> what remote repositories do we need, what are their contracts, and what are best practices for access?

---

## 1. remote repository: chutes hosted inference api

### 1.1 what is it?

chutes provides a **hosted inference api** at `https://llm.chutes.ai/v1/` that serves open-source language models via decentralized compute (bittensor subnet 64). this is the primary remote repository we need to access.

> "chutes supports all standard openai-compatible parameters" [1]

### 1.2 base url

| endpoint | url | purpose |
|----------|-----|---------|
| hosted inference | `https://llm.chutes.ai/v1/` | pre-hosted models, openai-compatible [1][2] |
| per-deployment | `https://{user}-{chute}.chutes.ai/v1/` | custom deployments [3] |

for `rhachet-brains-chutes`, we use the **hosted inference** endpoint — `https://llm.chutes.ai/v1/` — since we consume pre-hosted models, not custom deployments.

> the quickstart docs show per-deployment urls like `https://your-username-my-first-chute.chutes.ai` [3], but the llm-chutes plugin and litellm integration both use `https://llm.chutes.ai/v1/` for hosted model access [1][2].

### 1.3 authentication

| attribute | value |
|-----------|-------|
| env var | `CHUTES_API_KEY` [1] |
| key prefix | `cpk_` [4] |
| auth header | `Authorization: Bearer cpk_...` [4] |
| alt header | `X-API-Key: cpk_...` [1] |
| key creation | via chutes dashboard or cli: `chutes keys create --name yourkeyname --admin` [4] |

> "environment variable: `CHUTES_API_KEY`" — litellm docs [1]
>
> "chutes api keys follow the format `cpk_xxxxxxxxxxxxxxxx`" [4]

### 1.4 openai sdk compatibility

chutes is fully openai-compatible. the openai sdk works with a `baseURL` override:

```ts
import OpenAI from 'openai';

const client = new OpenAI({
  baseURL: 'https://llm.chutes.ai/v1/',
  apiKey: process.env.CHUTES_API_KEY,
});

const response = await client.chat.completions.create({
  model: 'Qwen/Qwen3-Coder-Next-FP8',
  messages: [{ role: 'user', content: 'hello' }],
});
```

> "chutes supports all standard openai-compatible parameters" — messages, temperature, top_p, max_tokens, frequency_penalty, presence_penalty, stop sequences, and tool/function call [1]

this is the same pattern used by `rhachet-brains-togetherai` — only `baseURL` and `apiKey` change.

### 1.5 supported operations

| operation | supported | endpoint |
|-----------|-----------|----------|
| chat completions | yes | `POST /v1/chat/completions` [3][4] |
| structured output (json_schema) | yes | via `response_format` param (vllm backend) [1] |
| tool/function call | yes | standard openai format [1] |
| stream | yes | `stream: true` param [4] |
| embeddings | yes | `/v1/embeddings` [1] |
| model list | yes | `GET /v1/models` [2] |

## 2. model catalog and model ids

### 2.1 model id convention

chutes uses **huggingface model identifiers** as model ids:

| format | example |
|--------|---------|
| `{org}/{model-name}` | `Qwen/Qwen3-Coder-Next-FP8` |
| `{org}/{model-name}` | `deepseek-ai/DeepSeek-R1` |
| `{org}/{model-name}` | `moonshotai/Kimi-K2-Instruct` |

## 3. rates and cost

### 3.1 chutes rates (confirmed)

| model | input per 1M | output per 1M | source |
|-------|-------------|---------------|--------|
| qwen3-coder-next | $0.07 | $0.30 | [5] |
| deepseek-r1 | $0.27 | $0.27 | [2] |
| deepseek-v3-0324 | $0.27 | $0.27 | [2] |

### 3.2 chutes vs together ai (cost comparison)

| model | chutes in/out | together ai in/out | delta |
|-------|--------------|-------------------|-------|
| qwen3-coder-next | $0.07 / $0.30 | $0.50 / $1.20 | **7x / 4x cheaper** |
| deepseek-r1 | $0.27 / $0.27 | $3.00 / $7.00 | **11x / 26x cheaper** |

> "chutes offers the best price for qwen3 coder next at the lowest rates: $0.070 per million input tokens and $0.300 per million output tokens" [5]

### 3.3 rate stability caveat

> chutes rates are subsidized by tao token emissions. if tao price drops, miners leave, and rates may change. rates should be treated as approximate and subject to change — unlike together ai's published rate card [6].

## 4. error modes

### 4.1 documented errors

| error | code | cause | user action |
|-------|------|-------|-------------|
| exhausted all available targets | 500 | chutes servers down | wait; check status page [7] |
| no instances available | 503 | model overload or restart cycle | wait several minutes [7] |
| invalid token | 401 | incorrect api key | verify key starts with `cpk_` [7] |
| model not found | 404 | misspelled or unavailable model | use valid model names from api [7] |

> "no instances available (503) is the key decentralization trade-off. when miner capacity for a model is exhausted, requests fail until miners come online." [6]

### 4.2 reliability profile

| attribute | together ai | chutes |
|-----------|-------------|--------|
| uptime guarantee | enterprise sla | none (miner-dependent) [6] |
| failure mode | queue/throttle | 503 "no instances" [7] |
| support | enterprise | community-driven [6] |

## 5. access pattern (best practices)

### 5.1 within this repo

the current `genBrainAtom.ts` pattern works identically for chutes — the only changes are:

| attribute | together ai | chutes |
|-----------|-------------|--------|
| `baseURL` | `https://api.together.xyz/v1` | `https://llm.chutes.ai/v1/` |
| env var | `TOGETHER_API_KEY` | `CHUTES_API_KEY` |
| `repo` | `'together'` | `'chutes'` |
| slug prefix | `together/` | `chutes/` |
| type name | `TogetherBrainAtomSlug` | `ChutesBrainAtomSlug` |

### 5.2 openai sdk pattern (industry standard)

both together ai and chutes use the same openai-compatible pattern. this is the industry standard for open-source inference providers:

```ts
const client = new OpenAI({
  baseURL: 'https://llm.chutes.ai/v1/',
  apiKey: process.env.CHUTES_API_KEY,
});
```

> litellm, openrouter, roo code, and other integrations all use the same openai-compatible pattern with `baseURL` override [1][8][9]

### 5.3 structured output

chutes supports `response_format: { type: 'json_schema', json_schema: { ... } }` via the vllm backend [1]. this is the same structured output mechanism used by `genBrainAtom.ts` today.

### 5.4 episode continuation

episode continuation (multi-turn via message history) works identically — the openai chat completions api is stateless; episodes are managed client-side by replay of the message array. no provider-specific logic needed.

## 6. lessons

### lesson 1: the access layer change is minimal

the entire remote access change is: `baseURL`, `apiKey` env var, and model config. the openai sdk, structured output, tool call, and episode continuation all work identically.

### lesson 2: model availability must be verified at runtime

unlike together ai (which publishes a static model catalog), chutes model availability depends on active miners. the `/v1/models` endpoint returns what's currently available. our config should document expected models, but consumers should be aware that 503 "no instances" is possible.

### lesson 3: rates need per-model research

chutes rates differ per model and are not published in a single rate card. each model's `BrainSpec.cost.cash` must be sourced individually — from the chutes dashboard, openrouter provider page, or pricepertoken.com.

### lesson 4: model ids use huggingface format

both together ai and chutes use huggingface-format model ids (e.g., `Qwen/Qwen3-Coder-Next-FP8`). however, exact id strings may differ (e.g., together ai may use `-tput` or `-FP8` suffixes that chutes doesn't). each model id must be verified against the chutes api.

### lesson 5: auth header format is standard bearer

chutes uses standard `Authorization: Bearer cpk_...` — same as together ai uses `Authorization: Bearer ...`. the openai sdk handles this automatically via the `apiKey` constructor param [1][4].

### lesson 6: no cache rate on chutes

like together ai, chutes does not publish separate cache hit/miss rates. `BrainSpec.cost.cash.cache` should remain `$0` for both get and set.
