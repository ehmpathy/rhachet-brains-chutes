# vision: rhachet-brains-chutes

## the outcome world

### before

you want to use cheap, powerful open-source brains via chutes. you have two options:

1. **raw openai sdk** — wire up `baseURL: 'https://llm.chutes.ai/v1'`, manage api keys, track token costs by hand, parse responses, handle errors, figure out model names. no type safety, no cost track, no episode management. every project reinvents the same plumb.

2. **rhachet-brains-togetherai** — same models, but 4-7x more expensive per token. together ai charges $0.50/$1.20 for qwen3-coder-next; chutes charges $0.07/$0.30. you pay the together ai premium for managed infrastructure you may not need.

### after

```ts
import { genBrainAtom } from 'rhachet-brains-chutes';

const atom = genBrainAtom({ slug: 'chutes/qwen3/coder-next' });
const { output, metrics } = await atom.ask({
  role: { briefs: [myBrief] },
  prompt: 'summarize this pull request',
  schema: { output: z.object({ summary: z.string() }) },
});

// output.summary = typed, validated, ready
// metrics.cash = $0.000042 (not $0.00024 like together ai)
```

same rhachet contract. same type safety. same metrics. 5-7x cheaper.

### the "aha" moment

you run `npm run test:integration` against chutes for the first time. the tests pass. the metrics show $0.003 total cost for 10 model calls. on together ai, the same suite cost $0.018. same models, same quality, fraction of the price. the rhachet contract didn't change — only the supplier did.

## user experience

### usecase 1: cost-sensitive agentic loops

**goal:** run hundreds of llm calls per task (code review, test generation, refactor) without budget blowout.

```ts
import { genBrainAtom } from 'rhachet-brains-chutes';

// qwen3-coder-next: $0.07/$0.30 per 1M tokens via chutes
// vs $0.50/$1.20 on together ai — 7x cheaper input, 4x cheaper output
const coder = genBrainAtom({ slug: 'chutes/qwen3/coder-next' });

const review = await coder.ask({
  prompt: `review this diff:\n${diff}`,
  schema: { output: z.object({ issues: z.array(issueSchema) }) },
  role: { briefs: [reviewerBrief] },
});
```

**timeline:** install package → set `CHUTES_API_KEY` → call `genBrainAtom` → done. same contract as `rhachet-brains-togetherai`, so migration is a package swap + env var change.

### usecase 2: multi-model selection

**goal:** pick the right brain for the task — cheap for bulk work, powerful for critical decisions.

```ts
import { getBrainAtomsByChutes } from 'rhachet-brains-chutes';

// get all available brains with their specs
const atoms = getBrainAtomsByChutes();

// each atom carries its BrainSpec — cost, context, swe-bench grade
// the orchestrator picks the best brain for each subtask
```

### usecase 3: episode continuation

**goal:** multi-turn conversations where context carries forward.

```ts
const atom = genBrainAtom({ slug: 'chutes/deepseek/v3.1' });

const first = await atom.ask({
  prompt: 'analyze this codebase structure',
  schema: { output: analysisSchema },
  role: {},
});

const second = await atom.ask({
  on: { episode: first.episode },
  prompt: 'now suggest refactor priorities',
  schema: { output: prioritiesSchema },
  role: {},
});
// second call sees the full conversation history
```

## mental model

### how would a developer describe this to a friend?

> "it's a drop-in adapter that lets rhachet talk to chutes instead of together ai. same open-source models, same api shape, but 5-7x cheaper because chutes runs on decentralized compute."

### analogy

chutes is to together ai what a wholesale club is to a retail store. same products, different supply chain, lower prices. `rhachet-brains-chutes` is the membership card — it plugs you into the wholesale supply chain through the same rhachet interface you already know.

### terms

| our term | their term | what it means |
|----------|------------|---------------|
| brain atom | model / endpoint | a single stateless inference call |
| slug | model id | `chutes/qwen3/coder-next` → `Qwen/Qwen3-Coder-Next` |
| brain spec | specs / rate | cost, context, swe-bench grade per model |
| episode | conversation | message history for multi-turn |
| brief | system prompt | role instructions loaded from artifacts |
| supplier | provider | the entity that supplies inference compute |

## evaluation

### how well does it solve the goals?

| goal | verdict |
|------|---------|
| cheap open-source inference | 5-7x cheaper than together ai for same models |
| fund open-source vision | chutes miners run open-source models; tao emissions incentivize open compute |
| variant intelligence | 10+ frontier models, same contract, swappable by slug |
| rhachet ecosystem fit | identical BrainAtom contract — genBrainAtom, ask(), BrainSpec, episodes |

### pros

- **5-7x cheaper** — qwen3-coder-next at $0.07/$0.30 vs $0.50/$1.20
- **same contract** — drop-in replacement for rhachet-brains-togetherai
- **open-source aligned** — decentralized compute, no vendor lock-in
- **full brain specs** — cost, context, swe-bench grades per model
- **type-safe** — zod-validated structured output, typed slugs

### cons

- **miner-dependent reliability** — 503 "no instances available" when miners drop off
- **no enterprise sla** — no guaranteed uptime, no contractual accountability
- **prompt visibility** — prompts visible to anonymous miners in cleartext (unless tee-enabled chutes)
- **crypto-dependent rates** — rates subsidized by tao token emissions; if tao crashes, miners leave
- **no managed fine-tune** — chutes offers self-managed docker deploys, not managed lora like together ai

### edgecases and pit of success

| edgecase | how we handle it |
|----------|-----------------|
| 503 "no instances available" | caller retries; BrainSpec documents reliability profile so orchestrators can make informed decisions |
| model name changes on chutes | slugs abstract over raw model ids; config update in one place |
| rate changes | BrainSpec holds cost per token; update config, all consumers see new costs in metrics |
| chutes goes down entirely | swap to `rhachet-brains-togetherai` — same models, same contract, different supplier |
| sensitive data concerns | document in BrainSpec or readme that prompts are visible to miners; users choose consciously |

### what's awkward

1. **the repo currently says "together" everywhere** — package.json name, slug prefixes, type names, env var. the wish is to replace all of this with chutes equivalents. this is the core work.

2. **chutes model catalog may differ from together ai** — not every together ai model is on chutes, and chutes may have models together ai doesn't. the config needs research and rebuild from chutes' actual catalog, not copy-paste from together ai.

3. **env var change** — `TOGETHER_API_KEY` → `CHUTES_API_KEY`. base url changes from `api.together.xyz/v1` to `llm.chutes.ai/v1`. auth key prefix is `cpk_` on chutes.

4. **rate research needed** — chutes rates differ from together ai. each model's BrainSpec cost needs to be sourced from chutes' actual rates, not assumed from together ai.

5. **reliability difference is material** — together ai has enterprise sla; chutes has miner-dependent availability. this should be clearly documented so consumers make informed choices, not surprised by 503s.
